# 謝罪会見ゲーム

本ゲームは、インシデント発生時の「会見」を体験するためのゲームです。

- 仕組みとしては、キャット＆チョコレート を利用しています。
- インシデントが記載されたイベントカードを引き、手持ちのアイテムを使って会見を乗り切る大喜利ゲームです。
- どれだけ説得力のある会見ができるかを競います。面白さを求めても構いませんが、会見を聞いた人がどれだけ納得できるかが評価点です。
- 他のプレイヤーが、会見を乗り切れたかを判定します。多数決で成功/失敗を判定します。全てのプレイヤーが会見をした後、全体での評価を話し合い、最優秀プレイヤーを決めます。

|項目|内容|
|:--:|:--|
|商品名|謝罪会見ゲーム インシデント報告会見編|
|プレイ人数|1-16人（1チーム1-4人、2-4チーム）|
|プレイ時間|30分（1ルーチン）|
|対象年齢|指定なし|
|ジャンル|教育、セキュリティ大喜利|
|初回リリース|2025-05予定|

## QuickStart

別途ページを作成予定。

## リソース

以下の4種類のカードを利用する。
1. 企業プロファイルカード（8枚）
2. インシデントカード（30枚）
3. 状況カード（20枚）
4. イベントカード（18枚）

- 企業プロファイルカード
  - ロールプレイ対象の企業プロファイルを示すカード
    - 従業員数や企業規模の組み合わせが示されている（大企業、スタートアップ、等）
  - 従業員数や業種や成熟度を提供することで、会見は誰に向けて行う事べきか、等を決る指標となる
- インシデントカード
  - 「何が」「どれだけ」起きたかを示す、インシデントを示すカード
    - ランサムウェアや内部不正、基幹系の停止や個人情報10万件流出、等
- 状況カード
  - ログやEDRやBCP等の備えの有無を示すカード
    - ログ保持90日、EDR未導入、机上演習未実施、等
- イベントカード
  - 不確実性を演出し、運要素を付与するカード
    - 漏洩データ公開等の負の要素、危機管理会社支援等の正の要素、正も負も影響しない要素、等
  - 現実世界でも、ログは機器上に残っていたが対策としてアップデートをしてしまい焼失した、等もある

カード定義は、[./cardList.md](./cardList.md)に配置している。

## ゲーム進行概要

全体の進行としては以下の通り

1. プレイヤーを集める
2. カードを準備する
3. ゲームをする（2回程度）
4. 得点の評価/表彰をする
5. 全体の講評をする

ゲーム部分に関しては、以下のような進め方になる

a. カードを引き、公開する
b. チーム内議論を行う
c. 全体に、会見と質疑応答を行い、得点を付ける
d. 今回の優秀チームを決め、得点を付ける

### 1.プレイヤーを集める

参加人数総数から、チーム数とチーム人数を決める。
- 独りで実施することもできるが、振り返り学習の多様性が生まれない為、別途他者との議論を行うことが望ましい
- ゲームとする場合は、最低2チームが必要となる
  - チーム人数は、インシデント対応経験の有無により、増減させる
  - チーム内人数が多ければ多いほど、議論ができなくなる可能性が高い
- チーム内で、リーダーと発表者を決める
  - リーダーは、カードを引いたり議論をまとめる
  - 発表者は、チーム内議論後に全体への会見を行う
  - 人数次第ではCISO等の立場を割り当てることで、ステークホルダー毎の思考をロールプレイしても良い

教育で使用する場合、1会全てを回す時間を考慮すると、4チーム程度が望ましいと思われる。

### 2.カードを準備する

1組のカードから、4つの種類のカードを分類してまとめてシャッフルし、場に配置する
- 企業プロファイルカード
- インシデントカード
- 状況カード
- イベントカード

### 3.ゲームをする

後述のゲーム進行を参考に、ゲームを進める

### 4.得点の評価/表彰をする

ゲームが終わったら得点の一番高いチームを優勝とする
- 必要に応じて、表彰や評価をする

### 5.全体の講評をする

ファシリテーターから、ゲーム全体を通したアドバイスや講評を行う
- ファシリテーターに知見が無い場合は、ゲームを通してどのような点を考慮する必要があったか、それはどのような意味があるのか、等を問いかける
- 全体講評をもってクローズとする

## ゲームモード

教育も考慮し、以下のゲームモードを用意している。基本的な進行方法は同じだが、カード情報を共有して利用するかなどの違いがある。

- ゲーミングモード
  - 基本モードとて、1組のカードを4チームで利用する
  - 1つの卓に4チームが付き、各チームごとにカードを引いていく
    - 同一空間で同じことをしている連帯感を感じつつ、与えられる状況も異なるので、ゲーム性がある
- 同一条件モード
  - 中央で代表者がカードを引き、全チームが同じ条件で会見を行う
  - 与えられる状況が同一であるため、論点や発表方法で差が出やすい
    - 何をどのように考え、どのように表現(会見)すべきか、を深く考えられるモード
- 競技モード
  - チームごとにカードを用意し、ゲームを行う
    - 各卓でユニークなシナリオが形成されるため、より採点が刺激的になる
- 学習モード
  - 企業プロファイルとインシデントカードだけ共通とし、各チームで状況カードとイベントカードを引く
  - この際に利用するカードは、ゲーミングモードと同様に、1つの卓で1組のカードを利用する
  - 大企業卓 vs 中小企業卓 のように分けても良い
    - 企業規模による対応考慮点の違いなどを学習可能

本マニュアルではゲーミングモードを中心に解説する。その他のモードに関しては、ファシリテーターが上記モード例を参考に自由に設定頂きたい。
- 今後、詳細な説明が追加される可能性がある

## ゲームの詳細

前述の通り、以下のルーティンを繰り返し、高得点を目指す

a. カードを引き、公開する
b. チーム内議論を行う
c. 全体に、会見と質疑応答を行い、得点を付ける
d. 今回の優秀チームを決め、得点を付ける

以下に、基本となる `ゲーミングモード` での進行方法を記載する

### ゲーミングモードでの前提

- 1卓に、4チームがアクセスできるように座る
- チームに於いて、`リーダー`と`発表者`の役割を決める
- 卓上に1組のカードを配置し、4チームが順番にカードを取得する
  - カード種類ごとに順番に引いて回る
  - これは状況に合わせて変えてよく、1チームが全種類のカードを引いてから次のチームが引く、でも良い

### a. カードを引き、公開する

各チームリーダーが、場からカードを引く

|カード種類|引く枚数|公開/非公開|
|:--|:--|:--|
|企業プロファイルカード|1|公開|
|インシデントカード|1|公開|
|状況カード|2|非公開|
|イベントカード|1|非公開|

`企業プロファイルカード`と`インシデントカード`を公開することで、

全チームが引き終わったら、次のフェーズに進む

### b. チーム内議論を行う

チーム内で`状況カード`と`イベントカード`を公開し、どのような会見をすべきか議論する
会見評価シートを基に、どこに向けた会見であるべきかなどを検討する。

全チームが並行して議論が進み、指定時間（標準は5分を想定）が過ぎたら次のフェーズに進む
- 教育上、議論を続けたほうが良い状況なら時間を延長する
  - その場合、全体としてゲームを回す回数が少なくなることを考慮する
- 時間を区切るのは、会見は外部の状況により早めなければいけない場合もあり、全ての議論を深くできるわけではないことを再現している

### c. 全体に、会見と質疑応答を行い、得点を付ける

チーム毎に会見と評価を行い、全チームの評価を行う。
評価方法は、会見評価シートを基に考える。

- 全チームに対し、順番に会見と質疑応答を行う
  - 会見は、1-2分程度で行う
  - 会見終了後、他のチームから質疑応答を受け付ける
    - 会見者は、与えられた情報とそこから推察できる情報で回答し、嘘をついてはいけない
    - 質疑を行う者は、どのステークホルダーの観点で考慮漏れが無いかという視点での質疑をし、相手を打ち負かす/貶める事を目的としてはならない
- 会見は以下の項目で各チームが評価し、理解がができると感じるチームが過半数の場合は `+1ポイント` を獲得する
  - 評価項目を参考に評価する  - 
  - あくまで、与えられた状況で善戦できたか、を評価軸と考えて投票する

### d. 今回の優秀チームを決め、得点を付ける

対応が良かったチームを多数決で選出し、追加点数を付与する。

- 全チーム発表後に最優秀チーム(善戦した、頑張った)を多数決で選出し、`+1ポイント`を獲得する
  - 結果の評価ではなく、状況が悪いが取りうる最善を尽くしたこと等を評価する
  - 評価対象チームがバラバラで多数決にならない場合は、合議をして決定するか、該当なしとする

## 会見評価シート

チーム議論や会見の評価をする際に、以下の内容で評価をする
- あくまで一例であり、観点の提供の意味合いであることに留意

|評価軸|0点|1点|2点|参考ガイドライン|
|:--|:--|:--|:--|:--|
|A.真実性|不正確/推測だらけ|情報は正確だがあいまいに終始|事実を正確に述べ、不明点は「未確認」と明示|NIST SP800-61 §3.2|
|B.状況把握の明確さ|何が起きたか不明瞭|インシデント概要のみ提示|発生原因・影響範囲・進捗等を数字で明示|GDPR Art33「事実・影響・対策の説明」|
|C.ステークホルダー配慮|特定集団を無視|一部のみ配慮|規制当局・顧客・従業員・投資家すべてに言及|MIT Sloan Review "Cyber Crisis Comms"|
|D.被害リカバリ施策|謝罪のみ/曖昧|金銭補償など言及|具体策と具体期日/窓口を提示||
|E.再発防止策の実現性|スローガンのみ|対策は提示するがあいまい|対策・責任者・ロードマップを明確化||
